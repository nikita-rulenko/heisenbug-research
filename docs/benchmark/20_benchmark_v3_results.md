# Benchmark v3: Scale Test (637 tests)

## Цель

Проверить гипотезу коллеги (Александр Карпушин): "Ты описал по мелкому проекту, где все помещается в контекст, а когда нет — тип Helixir выигрывает, но при этом статы нет."

Масштабирование: 175 тестов (v2) -> 637 тестов (v3), контекст 21KB -> 29KB.

## Методология

- Тот же протокол: generator != evaluator, 3 прогона, медиана
- Generator: Cerebras gpt-oss-120b
- Evaluator: Cerebras zai-glm-4.7 (reasoning model)
- Part A: 12 сценариев (192 max), Part B: 5 (80 max), Part C: 5 (80 max)
- **Важно**: Mem0 и Helixir хранят устаревший контекст (175 тестов), не обновлялись перед v3

## Анализ: почему v3 не показал ожидаемого расхождения

### 1. Контекст все ещё помещается в окно

29KB (~7K токенов) — это значительно меньше context window даже самых маленьких LLM (128K+). Гипотеза о том, что "MD файлы перестанут влезать" не подтвердилась на этом масштабе. Для реального расхождения нужен проект с контекстом 200-500K символов.

### 2. Устаревшие данные в памяти — главный инсайт

**Mem0 и Helixir не были обновлены** после масштабирования с 175 до 637 тестов. Это дало неожиданный, но ценный результат:

- **Mem0**: хранит "62 теста" и "175 тестов" — числа из старых сессий. На вопрос "сколько тестов?" отвечает неверно. Accuracy drop: 88.8% -> **58.8%** (-30pp)
- **Helixir**: каузальные цепочки содержат устаревшие связи. Потратил 170K токенов (3x больше MD), но восстановил только 75% контекста

### 3. Стоимость обновления памяти — скрытый расход

MD файлы / GitHub Issues обновляются бесплатно (перезапись файла). Mem0 и Helixir требуют:
- Переиндексации всех memories (~30+ записей в Mem0)
- Пересборки графа знаний в Helixir
- Каждое обновление — API вызовы к embedding модели

Это **скрытая стоимость владения**, не учтённая в v2.

### 4. GitHub Issues неожиданно вырос

+1.4% при увеличении масштаба. Гипотеза: формат issues (заголовок + body) лучше структурирует большие объёмы данных для LLM, чем плоский текст MD.

### 5. Graphiti — самый большой рост

+3.7% — граф-подобная структура ([Node: name]) помогает LLM лучше ориентироваться в растущем контексте.

## Выводы для доклада Heisenbug 2026

1. **На масштабе 29KB контекста подходы выравниваются** (72-76% total). Разница 3.5pp — статистически незначима при stddev 7-15.

2. **Семантическая память требует обслуживания**. Mem0 и Helixir проигрывают по accuracy recovery при устаревших данных. Это OPEX, который не виден при первоначальной настройке.

3. **Стоимость онбординга**: простые подходы (MD) — $0.032 за онбординг. Helixir — $0.102 (3.2x дороже) с худшим результатом (75% vs 88.8%).

4. **Для показа реального расхождения** нужен проект с контекстом, который физически не помещается в context window (~500K+ символов, ~125K+ токенов). На текущем масштабе (29KB) все подходы работают примерно одинаково.

5. **Самый ценный инсайт** — не "какой подход лучше", а **стоимость поддержания актуальности контекста**. MD файлы бесплатно обновляются, память — нет.

## Порог расхождения (уточнённая оценка)

| Размер контекста | Ситуация | Рекомендация |
|-----------------|----------|--------------|
| <50KB (~12K tok) | Всё в окне, подходы одинаковы | MD files (бесплатно, просто) |
| 50-200KB | Начинает не влезать, нужна выборка | GitHub Issues + поиск |
| 200-500KB | Точно не влезает, нужен RAG | Mem0 / Helixir (если обновлены) |
| >500KB | Enterprise scale | Graphiti / Helixir (каузальный граф) |
