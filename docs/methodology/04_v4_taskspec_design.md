# v4 Task-Spec benchmark — дизайн и обоснование

> **Статус:** дизайн зафиксирован, прогон готовится.
> **Дата:** 2026-04-17.
> **Зачем:** закрыть две самые дорогие проблемы v2/v3 — одинаковый контекст
> для всех подходов и gold-и, наказывающие фильтрацию.

## TL;DR

- Вместо того чтобы просить LLM **ответить на вопрос** с готовым контекстом,
  мы просим её **написать task-spec MD-файл** — техническое задание на тесты.
- Каждый из 5 подходов получает **свой** контекст (из своего источника
  памяти), а не одинаковый `context_text`.
- Затем другая модель (critic) читает эту spec в **изолированной сессии** и
  ищет **открытые вопросы** с оценкой критичности (1/2/3).
- Финальный скор подхода = **Σ criticality** найденных дыр. **Ниже — лучше.**
- Нет gold-а с перечислениями → нет штрафа за синтез и фильтрацию.
- Авторство идеи: Вячеслав Смирнов (куратор Heisenbug).

---

## Почему v4 вообще понадобился

v2/v3 ранжировали 5 подходов с разбросом 2–4% (72.4% у Helixir против 75.9%
у GitHub Issues). На конференции это подаётся как «выбор подхода важен».
Но в [`03_phase4_plan.md`](03_phase4_plan.md) мы сами зафиксировали, что
этот разброс во многом — шум LLM, а не различие retrieval-слоёв. Две
главные причины:

### Проблема 1. Одинаковый `context_text` для всех

`benchmarks/v2/runner.py:255-259` — одна и та же склейка episodes идёт во
все 5 прогонов. Разница между подходами = разница в том, как разные
прогоны LLM ведут себя на идентичном входе.

Честный замер retrieval есть только в [Context Recovery](../results/context_recovery.md)
— там каждый подход реально поднимает свой стек (Mem0 MCP, Helixir MCP,
GitHub MCP, `.cursor/rules` через файлы) и получает ответ на вопрос через
свой источник. Там и разрыв оказался реалистичным: 88.8% у MD/Issues/Graphiti
против 58.8% у Mem0.

### Проблема 2. Gold-и с перечислениями

S2, S4 из Part A требуют назвать 28 тестов и 70+ файлов. Подход, который
синтезировал осмысленный короткий ответ, получает 1–2 балла. Подход,
вывалившийся длинным списком из контекста, получает 4. Мы **измеряем
способность дампить контекст**, а не работать с ним.

v4 убирает оба фактора сразу.

## Формулировка задачи в v4

Вместо «ответь на вопрос» — «напиши MD-файл с постановкой задачи на тесты».

Пример входа planner-у:

```text
Тема: "Добавить систему купонов на скидку".
Твоя роль: senior test engineer.
Используя доступный тебе контекст проекта, напиши task-spec.md в формате:

# Task: <тема>
## Контекст
## Что нужно сделать
## Edge cases
## Границы задачи (что вне скоупа)
## Открытые вопросы (если есть)

Лимит: 500 слов. Если какой-то факт тебе неизвестен — явно пиши "неизвестно",
не додумывай.
```

Ключевое: **по 5 темам** мы получаем **5 MD-файлов от каждого подхода**,
всего 25 artifacts. Каждый подход работает со **своим** контекстом (свой
MCP/ретривал), и это честное различие.

---

## Как critic ищет дыры

Critic (другая LLM) получает task-spec и реальный контекст проекта, находит
открытые вопросы: противоречия в spec, упущенные edge cases, несформулированные
зависимости, выдуманные факты, отсутствие критериев приёмки.

Каждому вопросу ставится **criticality 1/2/3**:
- **3 (blocker)** — без ответа разработчик не сможет начать работу или
  сделает заведомо неправильно (нет критерия приёмки, противоречие в
  требованиях, выдуманный факт о проекте).
- **2 (significant)** — задачу можно начать, но вероятен rework (упущенный
  edge case, нет указания на связанные модули, неполные границы скоупа).
- **1 (minor)** — улучшение ясности (формулировка, небольшие неучтённые
  детали, отсутствие примеров).

Скор подхода по теме = `sum(criticality) across all open questions`.
**Ниже — лучше.** 0 — идеальная spec (мало реалистично). Средняя
ожидаемая — 3–10 на тему.

---

## Борьба со смещением оценки

Это самое важное место. Критик — тоже LLM, и у LLM есть известные
систематические bias (предпочитать многословные ответы, свой «стиль»,
первое в списке). Меры:

### 1. Разные семейства моделей на ролях

Только Cerebras (бюджетное ограничение), но разные семейства:

| Роль | Модель | Семейство |
|------|--------|-----------|
| Planner (пишет spec) | `qwen-3-235b-a22b-thinking-2507` | Qwen (Alibaba) |
| Critic (ищет дыры) | `zai-glm-4.7` | GLM (Z.AI) |
| Spot-checker (10% выборка) | `deepseek-r1-distill-llama-70b` | DeepSeek/Llama |

Spot-checker не оценивает все spec, а пересматривает случайные 10% на
предмет того, не «поехал» ли critic в стабильный bias (например, всегда
ставит 3 какому-то одному подходу). Мера согласованности — Spearman rank
correlation между основным критиком и spot-checker-ом. Порог ≥ 0.8.

### 2. Изолированные сессии critic-а

Каждый вызов critic-а — **свежий HTTP-запрос без предыдущих сообщений**.
Критик не накапливает контекст по другим подходам и не «знает», что
уже видел от этого же подхода. Это блокирует effects вроде «Mem0 был
хуже в прошлой сессии, значит и сейчас хуже».

### 3. Анонимизация

Critic видит только содержимое task-spec. Ни имени подхода, ни
метаданных (модели, времени генерации, ID подхода) в input-е critic-а
нет. В логи имена подходов попадают только после evaluation.

### 4. Randomized order

Для каждой темы порядок проверки 5 подходов рандомизируется. Нужно, чтобы
порядковый bias (critic чуть строже к 3-му и 4-му ответу в ряду —
наблюдали в v2) не коррелировал с именами подходов.

### 5. Two-pass critique

Критик сначала **находит** все открытые вопросы без оценки, потом
**в отдельном проходе** ставит criticality каждому. Это нужно, чтобы
оценка не подстраивалась под желаемую сумму.

### 6. Инверсия шкалы

**Ниже — лучше.** Нет «рейтинга качества», который подталкивает критика
к завышению хорошо написанных ответов. Каждый найденный open question
штрафует подход.

### 7. JSON-only output

Critic обязан вернуть JSON со списком вопросов и `criticality`. Никакой
прозы. Парсится регуляркой — упавший парсинг = инцидент для ручной
проверки, не тихий 0.

Всё это задокументировано в промте critic-а и в README запуска.

## Выбор тем (5–7 штук)

Критерий тем: они должны **реально требовать разных видов памяти**, чтобы
подходы могли отличиться. Черновой список:

1. **Купоны на скидку** — требует knowledge about pricing, discount rules,
   existing `Order.CalculateTotal`.
2. **Миграция NewsItem → Article** — большое файловое impact analysis,
   связь с Part A/S4.
3. **Добавить concurrent writes в SQLite** — требует memory о D3
   (flaky-тест, ICU) и D4 (почему SQLite).
4. **Раздел тестов по build tags** — требует S10 (оптимизация suite).
5. **Добавить soft-delete для Product** — связывает 4 слоя + миграции.
6. **Вынести auth middleware** — (опционально) требует architecture memory.
7. **Rate limiting на API** — (опционально) новый концепт, проверяет, как
   подходы связывают с существующим handler-слоем.

Финальный список — в `benchmarks/v4_taskspec/tasks.json`.

## Что сохраняется из v2/v3

- Те же **5 подходов** (md_files, github_issues, mem0, graphiti, helixir) —
  иначе сравнивать не с чем.
- Тот же project state (Bean & Brew, 637 тестов, те же docs).
- Тот же оценщик-семейство (`zai-glm-4.7`) — чтобы можно было сверить с
  v3 на одинаковых моделях.

Что **новое** в v4:

- Planner ≠ evaluator (разные семейства).
- Per-approach контекст (реальный retrieval, не один общий `context_text`).
- Нет перечислительных gold-ов.
- Инверсная шкала (Σ criticality, lower is better).
- Изоляция critic-сессии.
- Spot-checker на 10% для обнаружения dominant-critic bias.

## Финальная метрика

Для каждой темы `t ∈ {1..5}` и подхода `a`:

```
score(a, t) = Σ criticality(open_question) across critic(spec(a, t))
```

Для подхода a:

```
total(a) = Σ_t score(a, t)    # lower is better
mean(a)  = total(a) / 5
```

Также сохраняем:
- Количество вопросов по критичности (3/2/1) отдельно — чтобы отличить
  «один blocker» от «пять minor». Сумма не должна быть единственной
  метрикой.
- Rank correlation между основным критиком и spot-checker-ом.
- Длину spec в словах (для контроля: не штрафуется ли многословность).

---

## Что v4 **не закрывает**

- Малая выборка. 5 тем × 3 прогона = 15 замеров на подход, против 66 в v2
  (22 × 3). Статистической значимости всё ещё мало. Это Phase 5.
- Subjectivity критика. LLM-критик — не человек. Мы его мерим через
  spot-checker (Spearman ≥ 0.8), но фундаментально это remains LLM-judge.
- Не проверяет temporal update: как подход обрабатывает изменение проекта
  после индексации. Это отдельный experiment (staleness).

Эти ограничения будут зафиксированы в итоговом `v4_taskspec_results.md`
после прогона.
