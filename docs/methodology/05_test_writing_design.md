# Test-Writing Stage — дизайн и обоснование

> **Статус:** реализовано, дополнительная стадия для бенчмарка `context_recovery`.
> **Дата:** 2026-04-20.
> **Зачем:** проверить не только «что агент помнит», но и «может ли он применить запомненное к РЕАЛЬНОМУ коду» — написать работающий тест против реальной имплементации, без подсказок.
> **Не для демо со сцены.** Это easter egg для тех, кто заглянет в репозиторий после доклада.

## TL;DR

- После онбординга и Q&A-верификации агенту дают одну задачу: **написать новый Go-тест** для существующего метода в `coffee-portal`.
- Тест **реально пишется в репозиторий**, **реально запускается через `go test`**, и независимый судья (DeepSeek) оценивает результат.
- Запускается **до 3 топиков** на подход: **fixed** (одинаковый для всех 4 → прямое сравнение), **wide** (рандом из пула, seed=`hash(approach)`), **free** (агент сам выбирает метод).
- После каждого подхода — **полный откат репо** (`git reset --hard HEAD && git clean -fd`), чтобы следующий подход стартовал на чистом дереве.
- На дашборд эта стадия **не выводится** — Makefile-only.

## Зачем нужна отдельная стадия

Q&A-верификация (Phase 2) измеряет **точность извлечения фактов**. Это полезно, но не отвечает на главный вопрос конференции для тестировщиков: «*может ли агент**применить контекст для написания работающего теста*?»

Между «помнит, что метод `Order.CalculateTotal` принимает `[]OrderItem` со скидкой» и «написал валидный Go-файл который компилируется и проходит» — большой разрыв. Этот разрыв — и есть то, что мы измеряем здесь.

## Pipeline

Per `(approach, topic)`:

1. **plan** — planner LLM (`gpt-oss-120b`, Cerebras) получает онбординг-контекст подхода + бриф топика → отдаёт исходник Go-файла.
2. **write** — сохраняем в `coffee-portal/internal/<pkg>/zz_bench_<id>_<UUID>_test.go`. Префикс `zz_bench_*_test.go` — в `.gitignore` портала, двойная защита от случайного коммита.
3. **compile + run** — `go test -run ^TestUnitBench<UUID>$ -count=1 -timeout=60s -v ./...`. Запускается *full* `./...` — компилируются все пакеты, что позволяет детектить collateral damage (если LLM каким-то образом сломала компиляцию соседнего пакета).
4. **judge** — DeepSeek (`deepseek-chat`) оценивает результат по 4 критериям + отдельная метрика divergence.
5. **rollback** — `git reset --hard HEAD && git clean -fd` в портале. Безопасно потому что pre-bench состояние закоммичено.

## Топики и их ограничения

### Trade-off: выдуманные vs существующие фичи

Изначальная идея: «дать LLM написать тест к ЛЮБОЙ задаче, в т.ч. к фичам которых ещё нет в коде» (как в v4_taskspec). Отказались — потому что:

- Если фичи нет, LLM должна **написать имплементацию** вместе с тестом → это уже бенчмарк TDD-навыков, а не контекст-резолюшна.
- Сравнение становится несравнимым: «agent A написал импл+тест, agent B нашёл что фичи нет, отказался» — обоих как ранжировать?

**Решение:** все топики целятся в **существующие методы**. Имплементация уже есть, агент только пишет тест. Это узкая, но честная задача.

### Fixed topic
**`Order.CalculateTotal`** — хорошо описан в `test_context.json`, в Helixir-памяти, в MD-документах, в GitHub Issues. Информационная асимметрия минимальна → результат прямо отражает качество онбординга подхода, а не количество доступных данных.

### Wide pool (5 топиков)
Случайный выбор с seed = `hash(approach)`:
- `wide_news_summary_emoji` — `NewsItem.Summary` с CJK + emoji на границе maxRunes
- `wide_product_discount_boundary` — `Product.ApplyDiscount` boundary-кейсы
- `wide_category_validate_unicode` — `Category.Validate` с unicode/CJK
- `wide_order_can_cancel_matrix` — `Order.CanCancel` со всеми статусами
- `wide_news_validate_combos` — `NewsItem.Validate` edge-комбинации

В этих топиках информация в источниках неравномерна — какие-то детали могут быть в коде, но не в Helixir-памяти; какие-то — в MD, но не в Issues. **Это и есть стресс-тест**: насколько подход умеет давать LLM достаточно контекста, чтобы покрыть edge-cases.

### Free mode (агент выбирает сам)
Самый интересный режим. Бриф:

> «На основе онбординг-контекста, выбери САМ один существующий метод из `entity` для которого реально стоит написать дополнительный тест — реальный gap, а не очевидное место. Объясни выбор комментарием `// CHOICE: Type.Method — причина` в шапке.»

Никаких ground_truth_files на конкретный таргет — судье отдаём ВСЕ файлы пакета `entity` (агент мог выбрать любой). Судья парсит `// CHOICE:` чтобы понять что выбрал агент.

**Зачем нужен:** мы хотим увидеть не «насколько ты помнишь Х», а «насколько ты адекватен в принятии решений после онбординга». Подход с богатым reasoning-графом (Helixir) теоретически должен лучше различать «уже покрыто 14 subtest'ами» vs «реально не покрыто». Подход с плоской памятью (Mem0) скорее возьмёт что-то тривиальное.

**Что осознанно допускаем:** субъективную оценку выбора. Мы это явно фиксируем — `target_choice` это 5-й критерий, и DeepSeek даёт ему 1-4 на основании того, насколько по контексту реальной кодбазы выбор был неочевидным/осмысленным.

## Метрики

### 4 основных критерия для `fixed`/`wide` (1-4 каждый, max 16):
- **fact_grounding** — использует только реально существующие entity/методы/поля? (4 = всё реально, 1 = много выдумок)
- **convention_match** — соблюдает naming convention `TestUnitBench...` + table-driven + external `_test` package? (4 = полностью, 1 = совсем не похоже)
- **coverage_depth** — покрывает осмысленные edge-cases (boundaries, unicode, negatives)? (4 = глубоко, 1 = тривиально)
- **runs_clean** — компилируется и проходит? (4 = PASS, 3 = PASS с warnings, 2 = FAIL но логика разумная, 1 = compile error / panic)

### Дополнительный критерий для `free` (1-4, max 20):
- **target_choice** — насколько разумен ВЫБОР метода? 4 = неочевидный осмысленный gap, 3 = разумно но банально, 2 = слабый выбор, 1 = бессмысленный (например уже покрыто 14 subtest'ами)

### Отдельная метрика — divergence (0-4):
Сколько НЕсуществующих сущностей упомянуто в тесте (выдуманные методы, поля, константы, ошибки). 0 = ни одной выдумки, 4 = больше половины — выдумки.

**Гипотеза**: на широком топике divergence будет выше, чем на фиксированном — потому что агенту приходится домысливать в условиях неполного контекста. Эта дельта и есть полезный сигнал для сравнения подходов.

### Анти-bias
- Судья — `deepseek-chat` (DeepSeek), не Cerebras. Разные провайдеры → меньше скрытых корреляций между генератором и оценщиком.
- Промт включает **ground truth** (реальный код имплементации) — судья не угадывает, а сверяет с фактами.
- `runs_clean` — частично объективная метрика: exit_code от `go test` неоспорим.

## Безопасность

| Защита | Что делает |
|---|---|
| Pre-condition guard | `make _check-portal-clean` падает если portal не чист |
| `.gitignore` портала | Паттерн `zz_bench_*_test.go` блокирует случайный коммит |
| Уникальный UUID в имени | `secrets.token_hex(4)` — коллизий между параллельными прогонами не будет |
| Внешний пакет (`_test`) | Изолирован от внутренних test helpers |
| Per-topic rollback | `git reset --hard HEAD && git clean -fd` после каждого топика |
| `make scratch-clean` | Аварийная зачистка если процесс упал между write и rollback |

## Запуск

```bash
# Только эта стадия (контекст подхватывается из results/contexts/<approach>_latest.txt)
make bench-tw                              # все 4 подхода × оба топика
make bench-tw-approach APPROACH=helixir_mcp TOPIC=fixed
make bench-tw-approach APPROACH=mem0 TOPIC=wide

# Полный прогон: бенч + test-writing
make bench-full                            # все подходы, оба топика

# Подготовка
make portal-status                         # проверить что portal чист
make portal-baseline-tests                 # записать baseline `go test ./...`

# Аварийная зачистка
make scratch-clean
```

## Результаты

Сохраняются в `benchmarks/context_recovery/results/test_writing/<timestamp>_<approach>.json`. Содержат:
- исходник теста (что написала LLM)
- exit code и stdout/stderr `go test`
- raw-ответ судьи + распарсенные scores
- divergence отдельным полем

## Ограничения и что НЕ покрыто

- **Только entity-слой портала.** Топики целятся в `internal/entity/*`, не в usecase/handler/repository. Расширить — тривиально через WIDE_POOL.
- **Один судья.** Нет spot-checker'а на третьей семье моделей (как в v4_taskspec). Если будет интерес — добавить qwen-3 как контроль-судью на 10% сэмпле.
- **Нет per-criterion threshold'ов.** Сейчас просто сумма + divergence; можно добавить «runs_clean ≥ 3 обязательно для зачёта».
- **Контекст кешируется по умолчанию.** Если кодбаза изменится между `bench` и `bench-tw`, контекст будет stale. Опция: запускать через `--no-cache` или `make bench-full` (всегда fresh).
