# Phase 4 plan: честность бенчмарка и что мы доделываем до Heisenbug

> **Статус:** план, не запущено. Документ открыт для критики.
> **Дата:** 2026-04-17
> **Контекст:** на конференции практическая часть — ручная демонстрация adv_ промтов;
> сам бенчмарк не запускается со сцены. Но репозиторий публичный, и любой
> заинтересованный гость придёт и проверит. Этот файл фиксирует то, что мы
> сами видим как слабое место в v2/v3 и как мы это закрываем.

---

## TL;DR

Бенчмарки v2 и v3 измеряют **способность одной и той же LLM отвечать на вопросы
по одному и тому же тексту**, а не различия между подходами к памяти. Headline-
таблицы (`260 vs 255 vs 267`) находятся в пределах LLM-шума и не подтверждают
ни преимущества, ни недостатки конкретных подходов. Это нужно явно признать,
а не маскировать «победителями».

Phase 4 — это:

1. **Признать честно** ограничения v2/v3 в README и в самих файлах с
   результатами (правки уже намечены ниже).
2. **Запустить настоящий per-approach retrieval** через adv_ промты — там
   контекст действительно собирается инструментом подхода, а не подаётся
   готовым текстом одинаково для всех.
3. **Добавить anchor-baselines (floor + ceiling)** — без них любая разница
   между подходами не интерпретируется.
4. **Развести staleness как отдельный эксперимент**, а не как побочный эффект
   v3.
5. **Статистическая гигиена** — bootstrap CI, явное p-value, размер эффекта.

Все пункты ниже включают ссылки на код / данные, чтобы читатель мог проверить.

## 1. Что v2/v3 на самом деле измеряли

### 1.1. Один и тот же контекст для всех «подходов»

Файл: `benchmarks/v2/runner.py:255-259`

```python
def run_single(approach: str, context_text: str, run_id: int) -> RunResult:
    ...
    f"Here is the project context:\n\n{context_text}\n\n"
```

`approach` используется только как метка (имя файла результата + строка в
промте). Аргумент `context_text` собирается **один раз** до цикла
(`benchmark_runner_v2.py:392`) и **подаётся одинаковым LLM-у для всех
подходов**:

```python
context_text = "\n\n".join(f"### {ep['name']}\n{ep['content']}" for ep in episodes)
...
for i in range(num_runs):
    run = run_single(approach, context_text, i)   # тот же context_text
```

**Что это значит для Part A/B/C:**
- Никакого реального retrieval не происходит. Mem0 не делает `search_memory`,
  Helixir не строит граф, GitHub MCP не дёргает API. Результат — это
  способность gpt-oss-120b ответить по одному и тому же `episodes` тексту.
- Различия между «подходами» в Part A/B/C — это **семплинг-шум одной модели**
  на одинаковом входе с разными случайными seeds. Подтверждение: разброс
  между подходами (255–267 из 352, ≈3.4%) сопоставим с typical run-to-run
  variance gpt-oss-120b на тестах такого размера.

**Вывод.** Headline-таблица v3 («GitHub Issues 75.9% vs Helixir 72.4%»)
**не доказывает** превосходство GitHub Issues. Это сравнение требует либо
переинтерпретации, либо повторного прогона с настоящим retrieval (см. 4.1).

### 1.2. Что Context Recovery Benchmark действительно измеряет

`benchmark_context_recovery.py` — единственный из v3-сценариев, где между
подходами есть **реальная разница в источнике**. MD читаются с диска, Mem0
дёргается через MCP, Helixir строит граф. Здесь различия более значимы и
интерпретируемы:

- Mem0 58.8% — это про **staleness** (см. 1.4), а не про подход как таковой.
- Helixir 75% при 170K токенов — это про **онбординг-цепочку из множества
  tool-calls**, и здесь стоимость реальна.

Этот блок мы оставляем, но переименовываем секцию заголовка в репорте:
«Context Recovery — единственный сценарий с реальным per-approach retrieval».

### 1.3. Gold standards систематически наказывают фильтрующие подходы

Файлы: `benchmarks/shared/data/test_context.json`, `benchmarks/shared/data/decision_context.json`.

Часть gold-ответов перечисляют все 28 тестов или 70+ файлов (S2, S4). Подход,
который выдал семантически корректный, но сжатый список («все handler-тесты
+ middleware»), получает 0 баллов от evaluator-а. Подход, который вывалил
весь список целиком, получает максимум.

Это смещает результат в пользу подходов, оптимизированных на «вернуть весь
ингестированный текст», и против фильтрующих/обобщающих подходов
(семантический поиск Mem0, FastThink Helixir).

**Что делать:** в Phase 4 переписать ~30% gold-ов в формате
«acceptance criteria» вместо «exact list» — оценивать содержание, а не
полноту перечисления. Эталон — формат `pytest`-моделирования: «должно
быть упомянуто X из Y категорий», а не «должны быть перечислены все 28
имён».

### 1.4. Part C — это recall, а не reasoning

Все 5 подходов в Part C v3 получают 75–79 из 80 (93.8%–98.8%). Сценарии
Part C построены так, что gold-ответ дословно перефразирует фрагмент
ингестированного контекста. Это измеряет память LLM на 7K-токенный вход,
а не способность подхода поддерживать reasoning.

**Что делать:** в Phase 4 добавить Part D — counterfactual reasoning:
вопросы, ответ на которые **не содержится** в ингестированном тексте, но
выводится из него (например: «если убрать middleware X, какие тесты
сломаются?»). Это тестирует именно способность подхода поддержать вывод,
а не извлечение.

### 1.5. Staleness в v3 запутан с подходом

В v3 Mem0 и Helixir содержали данные с 175 тестов, MD/GitHub/Graphiti были
обновлены до 637. В таблице это выглядит как «MD/GitHub стабильнее», но
на самом деле смешаны два эффекта: (а) свойство хранилища обновляться, (б)
факт того, что свежие данные были у одних и не было у других.

Корректный эксперимент: либо все подходы со свежими данными, либо все со
устаревшими — два прогона, и delta показывает влияние staleness на каждый
подход отдельно.

### 1.6. Нет floor / ceiling

Без двух якорей — «глупого» (только заголовки файлов, no context) и
«идеального» (полный код проекта в контексте) — невозможно сказать, насколько
75% — это «хорошо». Может быть, floor уже даёт 65%, а ceiling — 95%, и весь
разброс между подходами лежит в нижней половине этой шкалы.

### 1.7. Статистическая значимость

В `12_benchmark_design.md` явно написано: «при N=3 различия в пределах
~5% не значимы». Тем не менее в headline-таблицах v3 обсуждается delta
+1.4% и -2.8% как реальные. Это противоречие нужно убрать: либо повысить N,
либо в каждой таблице писать CI и явно отмечать «within noise».

## 2. Что мы делаем в Phase 4

### 2.1. Real-retrieval run через adv_ промты

**Где:** `prompts/adv_onboarding_*.md` в `heisenbug-coffee-portal`.

**Что:** для каждого из 4 подходов прогнать adv-промт в Cursor/Claude Code,
зафиксировать:

- список tool-calls подхода (Mem0 search/add, Helixir search/think, GitHub
  list_issues/get_issue, MD read);
- wall-clock по фазам;
- output, разделённый на BLOCK 1 (общий) / BLOCK 2 (что дал именно этот
  инструмент) / BLOCK 3 (метрики).

**Что это даёт:** впервые получаем числа, где между подходами есть
**разница в источнике контекста**, а не одинаковый text-blob. Это и есть
честный сравнительный замер.

**Acceptance:** 4 заполненных отчёта (по одному на подход), результаты
сведены в новую таблицу в `20_benchmark_v3_results.md` с пометкой
«Phase 4.1 — real retrieval».

### 2.2. Floor + Ceiling baselines

**Floor:** прогон с контекстом «список файлов проекта без содержимого»
(~2KB). Что LLM может ответить на Part A/B/C, опираясь только на структуру?

**Ceiling:** прогон с полным `internal/**/*.go` в контексте (~150KB).
Что LLM выжмет, имея вообще всё?

**Что это даёт:** интерпретационную шкалу. Если floor = 60% и ceiling =
85%, то разброс подходов 72–76% занимает 50% этой шкалы — есть что
обсуждать. Если разброс 72–76% при floor = 70% и ceiling = 78%, разговор
закрыт — все подходы около floor, разница не значима.

**Acceptance:** 2 строки в headline-таблице, добавленные с пометкой
`[anchor]`.

### 2.3. Staleness как отдельный эксперимент

Два прогона на одном и том же подходе:

- **Fresh:** память синхронизирована с проектом (637 тестов).
- **Stale:** память содержит «172 тестов», как было в v2.

Delta = «штраф подхода за устаревание». Это число — реальный показатель
качества подхода в долгосрочной эксплуатации.

Прогнать для всех 4 подходов. Ожидание: MD/GitHub имеют delta ≈ 0
(перечитываются каждый раз), Mem0 — большая (dedup wall), Helixir —
средняя (есть update_memory, но цепочки ссылаются на старые узлы).

**Acceptance:** таблица «Staleness penalty per approach» в новом разделе
`20_benchmark_v3_results.md`.

### 2.4. Scale-up — отложено

В оригинальном гипотезе Александра Карпушина («контекст не помещается»)
разница должна проявляться на 200–500K. v3 (29KB) этого не достигает.

Запуск на 500KB контекста требует генерации соразмерного синтетического
проекта — это работа, не вписывающаяся в pre-Heisenbug сроки.

**Решение:** пометить как Phase 5, в докладе явно сказать «гипотезу о
расхождении на больших контекстах мы не закрыли — это next step».

### 2.5. Статистическая гигиена

- Поднять N с 3 до 10 для финальных headline-цифр.
- Bootstrap 95% CI на каждой ячейке.
- В каждой сравнительной таблице добавить колонку «within-noise?» (Y/N).
- Заголовки типа «GitHub Issues выигрывает» убрать в пользу «в пределах
  шума, см. CI».

**Acceptance:** обновлены `16_benchmark_results.md` и
`20_benchmark_v3_results.md`. Таблицы с CI или явной пометкой «N=3,
ориентировочно».

## 6. Критика этого плана

Этот документ намеренно самокритичен. Есть риски и в обратную сторону:

- **Слишком жёсткая самокритика подрывает доверие к доклaду.** Контр-аргумент:
  доклад про context degradation, метакритика собственной методологии — это
  тематически уместно и работает на основной тезис.
- **«Признали — значит можно было не делать v2/v3».** Нет: v2/v3 дали
  Context Recovery (ценный замер), показали dedup wall практически (см.
  `21_practical_verification.md`), и без них мы бы не сформулировали
  Phase 4. Это итерация, а не ошибка.
- **Phase 4.1 (manual run adv_) — это не «бенчмарк», это «демонстрация».**
  Согласен. В докладе это позиционируется именно как демонстрация
  per-approach различий, не как статистический замер. Для статистики
  нужен Phase 5 + scale.
