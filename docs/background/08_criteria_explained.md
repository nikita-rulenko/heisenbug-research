# Критерии сравнения — расшифровка к таблице №10

Комментарий к слайду "Heisenbug 2026 — Контекст AI-агентов: 4 подхода".

---

## 1. Бесплатность

Можно ли использовать решение полностью бесплатно, без подписок и без скрытых расходов на LLM-вызовы.

- **MD файлы** ★ — абсолютно бесплатно: текстовые файлы в Git, никаких API.
- **GitHub Issues MCP** ~ — Free tier GitHub (50 premium requests/month). Для серьёзного использования нужен Pro ($10/mo).
- **Mem0** ~ — OSS-версия бесплатна, но каждый `add()` вызывает LLM (GPT-4 по умолчанию), а это токены. Cloud: Free до 10K memories, далее $19-249/mo.
- **Helixir** ~ — OSS бесплатен, но для работы **обязателен LLM** (extraction + dedup). `HELIX_LLM_API_KEY` — required env variable. Можно использовать Cerebras (free tier) или Ollama (local), но LLM всегда в pipeline.

## 2. Без LLM-вызовов в pipeline

Требуются ли LLM-вызовы для базовых операций записи и чтения контекста. Критерий важен для стоимости, latency и privacy.

- **MD файлы** ★ — чистый текст, LLM не участвует вообще.
- **GitHub Issues MCP** ★ — MCP делает REST-вызовы к GitHub API, LLM не нужен для хранения/чтения.
- **Mem0** ✗ — каждый `add()` отправляет беседу в LLM для extraction фактов + embedding. Это архитектурное решение: «умная» память, но за каждую запись платишь токенами.
- **Helixir** ✗ — LLM обязателен. `add_memory` вызывает LLM для atomic fact extraction и smart deduplication (ADD/UPDATE/SUPERSEDE/NOOP). `HELIX_LLM_API_KEY` — required. Разница с Mem0: Helixir может использовать Cerebras (~3000 tok/s, free tier) или Ollama (полностью локально), но pipeline без LLM не работает.

## 3. Семантический поиск

Может ли система найти релевантные воспоминания по смыслу, а не по точному совпадению текста. Например, запрос "что пользователь ест" найдёт факт "вегетарианец, избегает молочных продуктов".

- **MD файлы** ✗ — только полнотекстовый поиск (grep/ctrl+F). Нет embeddings.
- **GitHub Issues MCP** ✗ — GitHub Search API ищет по ключевым словам, не по семантике.
- **Mem0** ★ — основа архитектуры. Факты хранятся как embeddings в vector DB (Qdrant/Pinecone/Chroma). Самый зрелый semantic search в категории.
- **Helixir** ✓ — SmartTraversal комбинирует vector similarity с graph traversal через HelixDB.

## 4. Граф связей

Хранит ли система связи между сущностями (пользователь → проект → технология → проблема). Графы дают контекст, которого нет в плоском списке фактов.

- **MD файлы** ✗ — плоский текст, связи только в голове у человека.
- **GitHub Issues MCP** ✗ — issues плоские, можно ссылаться друг на друга (#123), но это не граф.
- **Mem0** ✓ — Graph Memory (через Neo4j): извлекает entities и relationships из бесед.
- **Helixir** ★ — граф нативен (HelixDB = graph-vector DB). Каждый факт — узел, связи — типизированные рёбра. Это фундамент архитектуры, а не дополнение.

## 5. Каузальные цепочки

Может ли система строить и хранить причинно-следственные связи: "A привело к B", "C потому что D", "E противоречит F". Это уникальная возможность для reasoning — агент не просто помнит факты, а понимает почему.

- **MD файлы** ✗ — человек может описать причины в тексте, но система не парсит их.
- **GitHub Issues MCP** ✗ — нет механизма каузальных связей.
- **Mem0** ✗ — граф хранит entity relationships (User → works_at → Company), но не каузальные (A → BECAUSE → B).
- **Helixir** ★ — единственное решение с нативными каузальными типами: IMPLIES, BECAUSE, CONTRADICTS, SUPERSEDE. Агент может пройти по цепочке причин и понять контекст решения.

## 6. Авто-извлечение фактов

Может ли система сама определить, что из беседы стоит запомнить, без явной команды от пользователя.

- **MD файлы** ✗ — человек вручную пишет и обновляет файлы.
- **GitHub Issues MCP** ✗ — человек вручную создаёт issues.
- **Mem0** ★ — главная фишка. LLM (GPT-4) анализирует беседу и сам извлекает: "пользователь предпочитает async/await", "работает на TypeScript". Дедупликация и обновление автоматические.
- **Helixir** ★ — аналогично Mem0: `add_memory` запускает LLM-powered "Atomic Fact Extraction" — декомпозицию текста в атомарные факты + "Smart Deduplication" (ADD/UPDATE/SUPERSEDE/NOOP). Cognitive Protocol добавляет автоматические триггеры (auto_recall при старте, auto_save при новых фактах).

## 7. Working memory (scratchpad)

Есть ли у агента изолированное пространство для промежуточных рассуждений, которое не загрязняет долговременную память. Аналог: мы пишем на черновике, но не всё с черновика переносим в тетрадь.

- **MD файлы** ✗ — нет концепции рабочей памяти.
- **GitHub Issues MCP** ✗ — нет.
- **Mem0** ✗ — всё, что извлечено, идёт в постоянное хранилище.
- **Helixir** ★ — FastThink: изолированный scratchpad для reasoning. Результаты живут отдельно от основной памяти. При timeout — сохраняются с маркером [INCOMPLETE].

## 8. Онтология фактов

Типизирует ли система факты по категориям. Без типизации все воспоминания — плоский список. С типизацией агент знает: это навык? предпочтение? цель? мнение?

- **MD файлы** ✗ — свободный текст, структура на усмотрение автора.
- **GitHub Issues MCP** ✗ — labels дают базовую категоризацию (bug/feature), но это не онтология фактов.
- **Mem0** ✗ — flat memories: все факты хранятся как строки без типизации.
- **Helixir** ★ — 7 онтологических типов: skill, preference, goal, fact, opinion, experience, achievement. Агент может запросить "все цели пользователя" или "все навыки".

## 9. Temporal filtering

Может ли система фильтровать воспоминания по давности. Важно для релевантности: факт 2-летней давности может быть устаревшим.

- **MD файлы** ✗ — файл либо читается целиком, либо нет. Git blame даёт историю, но агент её не использует.
- **GitHub Issues MCP** ~ — GitHub API поддерживает фильтры по дате (created, updated, since). Базовый temporal filtering.
- **Mem0** ~ — через metadata можно добавить timestamps, но нет встроенных временных режимов. Auto-forget работает через LLM dedup.
- **Helixir** ★ — 4 встроенных режима: 4 часа (горячий контекст), 30 дней (рабочий), 90 дней (проектный), all (вся память). Агент выбирает режим в зависимости от задачи.

## 10. Multi-user isolation

Может ли система изолировать контекст разных пользователей. Критично для production-приложений, где один агент обслуживает множество людей.

- **MD файлы** ✗ — файлы общие для всего проекта, нет per-user контекста.
- **GitHub Issues MCP** ★ — нативная изоляция: разные repos, assignees, permissions. GitHub RBAC из коробки.
- **Mem0** ★ — `user_id`, `agent_id`, `session_id` — трёхуровневая иерархия. Лучшая в категории multi-tenant архитектура.
- **Helixir** ✗ — v0.1, multi-user пока не реализован.

## 11. MCP (Cursor)

Есть ли MCP-сервер для интеграции с Cursor (и другими MCP-compatible IDE). Это обязательное условие для демо.

- **MD файлы** ✓ — Cursor нативно читает файлы из repo. MCP не нужен — файлы уже в контексте.
- **GitHub Issues MCP** ✓ — официальный GitHub MCP Server (GA), OAuth 2.1.
- **Mem0** ✓ — Mem0 MCP Server для Cursor / Claude Desktop.
- **Helixir** ✓ — нативный MCP-сервер на Rust, часть проекта.

Все 4 решения интегрируются с Cursor — это уравнивает их в плане demo-ready.

## 12. Production-ready

Можно ли использовать в production прямо сейчас с гарантиями стабильности.

- **MD файлы** ★ — стандарт де-факто. AGENTS.md используется повсеместно. Ничего не может сломаться.
- **GitHub Issues MCP** ★ — GitHub MCP Server в GA с сентября 2025. Продукт GitHub, 15M devs.
- **Mem0** ★ — production: SOC 2 Type II, GDPR, $24M funding, 50K+ developers. Enterprise tier.
- **Helixir** ✗ — v0.1.1, 74 stars, 1 maintainer. Research-level проект, не для production.

## 13. Privacy / data sovereignty

Можно ли гарантировать, что данные не покидают контур организации.

- **MD файлы** ★ — всё в Git repo. Если repo приватный — полная изоляция.
- **GitHub Issues MCP** ✗ — данные на серверах GitHub (US/EU). Для regulated industries это может быть проблемой.
- **Mem0** ~ — Cloud: данные на серверах Mem0 (US) + проходят через LLM провайдера. OSS: полный контроль, но нужен self-hosted LLM (Ollama) для полной изоляции.
- **Helixir** ~ — HelixDB и MCP-сервер полностью self-hosted. Но LLM обязателен: с Ollama — полная изоляция, с Cerebras/OpenAI — данные проходят через LLM API.

## 14. Community / ecosystem

Размер сообщества, количество интеграций, доступность помощи и документации.

- **MD файлы** ✓ — огромная неформальная экосистема. Каждый проект с AI использует .md файлы. Но нет единого стандарта.
- **GitHub Issues MCP** ★ — 15M developers на GitHub, лучшая документация, официальная поддержка.
- **Mem0** ★ — 47.8K GitHub stars, $24M raised, интеграции с CrewAI, LangChain, AWS Agent SDK, Vercel AI SDK. Самая развитая экосистема среди memory frameworks.
- **Helixir** ✗ — 74 stars, 1 maintainer. Документация минимальна.
